## 5. The Shipping Spine (Operator Chain)
*End-to-end delivery chain from topic to merge.*

`TOPIC.md` тЖТ `PLAN.md` тЖТ `storage/dp/intake/DP.md` тЖТ execution тЖТ `RESULTS` + `CLOSING` тЖТ audit bundle тЖТ merge

* **Topic:** `storage/handoff/TOPIC.md` тАФ latest-wins planning input.
* **Plan:** `storage/handoff/PLAN.md` тАФ latest-wins planning output, draft input.
* **Active DP draft:** `storage/dp/intake/DP.md` тАФ latest-wins operator surface.
* **Packet/process identity:** `DP-OPS-XXXX` тАФ retained in the TASK leaf chain for base packets, addendum lineage leaves for addenda, certify receipts, and telemetry. Draft dispatch allocates the next packet id from the current certified TASK packet id plus one.
* **Certify:** `ops/bin/certify` тАФ generates `RESULTS`, emits SoP/PoW/TASK archive leaves, requires `STELA_TRACE_ID` from latest OPEN artifact.
* **Audit bundle:** `storage/handoff/AUDIT-*.txt` (initial) / `storage/handoff/AUDIT-R*.txt` (rerun) тАФ generated by `ops/bin/bundle --profile=audit`.
* **Merge:** Operator commits work branch, opens PR per `CLOSING` sidecar, merges to main.
* **Secondary lanes:** `foreman/addendum` (intervention), `conform/conformist` (normalization), `execution-decision` (disposable/manual) тАФ bounded sidepaths, not replacements for RESULTS or audit truth.
* **Delivery mechanics:** `docs/MANUAL.md`.
