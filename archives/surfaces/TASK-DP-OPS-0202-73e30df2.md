## 3.4 Execution Plan (A-E)

### 3.4.1 State
- Certify currently replays broad receipt commands exactly as written, which means a packet can trigger direct smoke, then trigger the same smoke again through `verify --mode=full`, even when most lanes are unrelated to the packet scope.
- That behavior inflates wall time, floods operator-visible `storage/` roots with transient smoke outputs during long runs, and obscures packet proof under duplicated repo hygiene churn.
- `DP-OPS-0200` improved determinism by giving smoke tests unique real output paths, but that same fix made the operator-visible churn obvious.
- The next gain is not more smoke coverage. It is lane specificity, quarantine, truthful receipt boundaries, and hard review of why a given lane exists inside certify at all.

### 3.4.2 Request
- Make certify replay the lanes the active packet actually needs.
- Keep full repo verify as a separate explicit tool, not a hidden repeated burden inside every certify.
- Move smoke outputs into quarantined runtime roots so the operator sees final artifacts, not fixture churn.
- Ensure RESULTS, telemetry, and audit evidence all say exactly what ran.
- Build explicit CbC review weight into the mechanism: retained certify lanes must declare owner, reason class, and proportion justification so future packets do not silently re-bloat replay by convenience.
- Generate RoR CbC decision leaves for the certify-lane review outcomes so the keep/defer/remove record survives beyond this packet and can be queried later.
- Add an infrastructure-importance column or adjacent canonical field in the existing registries for the certify-relevant binaries, lints, and tests so packet-local replay policy can consume a real canon importance signal such as `critical`, instead of inferring priority from convenience.
