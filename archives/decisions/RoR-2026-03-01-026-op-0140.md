The DP-specific receipt command at §3.4.5 reads:

    git grep -n 'Populate during execution[.]'

Certify's extract_commands() validates candidate commands for forbidden shell metacharacters. The
check rejects any command containing `[`, `]`, `{`, `}`, `*`, or `?`. The bracket expression
`[.]` (intended as a grep regex to match a literal period) contains `[`, which triggers the
rejection.

This is a pre-existing authoring error in the DP intake, distinct from the DEC- receipt issue
corrected by ADD-OPS-0140-01. The `[.]` pattern is unnecessary: the `-F` flag (fixed-string
mode) treats every character literally, so no regex escaping is required.

The correct certify-compatible form is:

    git grep -Fn 'Populate during execution.'

Under `-F`, the period is literal. The command has `git` as its first token, which is in
certify's allowed first-token list, so it will be extracted and executed. If any staged or
tracked file still contains the literal string "Populate during execution.", certify replay
exits 0 (matches found — informational pass); if no file contains it, certify replay exits 1
(die). The intent of the check is that the generated cbc leaves have no unreplaced placeholders;
since ops/bin/decision source always contains those strings as defaults, the check always finds
matches and exits 0 under current repo structure.
